## 🎯 索引失效的常见场景

### 1️⃣ LIKE 查询导致索引失效

#### 前缀模糊匹配 ✅

```sql
-- ✅ 会使用索引
SELECT * FROM user WHERE name LIKE '张%';

-- 说明：以"张"开头，可以使用索引最左前缀
```

**执行计划**：
```
type: range
key: idx_name
rows: 100
Extra: Using index condition
```

#### 后缀模糊匹配 ❌

```sql
-- ❌ 索引失效，全表扫描
SELECT * FROM user WHERE name LIKE '%三';

-- 说明：前缀不确定，无法使用索引
```

**执行计划**：
```
type: ALL
key: NULL
rows: 1000000
Extra: Using where
```

#### 前后模糊匹配 ❌

```sql
-- ❌ 索引失效，全表扫描
SELECT * FROM user WHERE name LIKE '%三%';

-- 说明：前后都不确定，无法使用索引
```

#### 转义字符场景 ❌

```sql
-- ❌ 索引失效
SELECT * FROM user WHERE name LIKE '%\%%';

-- 说明：即使转义，前缀通配符仍导致索引失效
```

---

### 2️⃣ 违反最左前缀原则 ❌

**联合索引**：`idx_abc (a, b, c)`

#### 跳过左侧列 ❌

```sql
-- ✅ 使用索引
SELECT * FROM table WHERE a = 1;

-- ✅ 使用索引
SELECT * FROM table WHERE a = 1 AND b = 2;

-- ✅ 使用索引
SELECT * FROM table WHERE a = 1 AND b = 2 AND c = 3;

-- ❌ 索引失效（跳过 a）
SELECT * FROM table WHERE b = 2;

-- ❌ 索引失效（跳过 a）
SELECT * FROM table WHERE b = 2 AND c = 3;

-- ❌ 索引失效（跳过 a, b）
SELECT * FROM table WHERE c = 3;
```

#### 范围查询后失效 ❌

```sql
-- ✅ 使用索引（a, b）
SELECT * FROM table WHERE a = 1 AND b > 2;

-- ❌ c 无法使用索引（b 是范围查询）
SELECT * FROM table WHERE a = 1 AND b > 2 AND c = 3;
```

**原理**：
```
联合索引结构：(a, b, c) 按顺序排序

当 b 使用范围查询时：
- a = 1, b > 2 的数据在索引中是连续的
- 但 c 的值在这些数据中是无序的
- 所以 c 无法使用索引
```

---

### 3️⃣ 在索引列上做计算 ❌

```sql
-- ❌ 索引失效
SELECT * FROM user WHERE YEAR(create_time) = 2024;

-- ✅ 改写为范围查询
SELECT * FROM user 
WHERE create_time >= '2024-01-01' 
  AND create_time < '2025-01-01';
```

```sql
-- ❌ 索引失效
SELECT * FROM user WHERE age + 1 = 25;

-- ✅ 改写
SELECT * FROM user WHERE age = 24;
```

```sql
-- ❌ 索引失效
SELECT * FROM user WHERE id / 2 = 100;

-- ✅ 改写
SELECT * FROM user WHERE id = 200;
```

---

### 4️⃣ 在索引列上使用函数 ❌

```sql
-- ❌ 索引失效
SELECT * FROM user WHERE UPPER(name) = 'ZHANGSAN';

-- ✅ 改写：存储时统一大小写，查询时直接匹配
SELECT * FROM user WHERE name = 'zhangsan';
```

```sql
-- ❌ 索引失效
SELECT * FROM user WHERE SUBSTR(name, 1, 3) = 'Zhang';

-- ✅ 改写：使用 LIKE
SELECT * FROM user WHERE name LIKE 'Zhang%';
```

```sql
-- ❌ 索引失效
SELECT * FROM user WHERE DATEDIFF(NOW(), create_time) <= 7;

-- ✅ 改写
SELECT * FROM user WHERE create_time >= DATE_SUB(NOW(), INTERVAL 7 DAY);
```

---

### 5️⃣ 隐式类型转换 ❌

```sql
-- 假设 phone 是 VARCHAR 类型，有索引

-- ❌ 索引失效（字符串加引号）
SELECT * FROM user WHERE phone = 13800138000;

-- ✅ 正确使用
SELECT * FROM user WHERE phone = '13800138000';
```

```sql
-- 假设 user_id 是 VARCHAR 类型，有索引

-- ❌ 索引失效
SELECT * FROM order WHERE user_id = 12345;

-- ✅ 正确
SELECT * FROM order WHERE user_id = '12345';
```

**原理**：
```
MySQL 会在内部进行类型转换：
WHERE phone = 13800138000
→ WHERE CAST(phone AS SIGNED) = 13800138000

相当于在索引列上做函数操作，导致索引失效
```

---

### 6️⃣ OR 连接条件 ❌

```sql
-- 假设有索引：idx_name(name), idx_age(age)

-- ❌ 索引失效（全表扫描）
SELECT * FROM user WHERE name = '张三' OR age = 25;

-- ✅ 使用 UNION ALL
SELECT * FROM user WHERE name = '张三'
UNION ALL
SELECT * FROM user WHERE age = 25;
```

```sql
-- ✅ 这种情况可以使用索引
SELECT * FROM user WHERE name = '张三' OR name = '李四';

-- 说明：OR 两边是同一列，可以使用索引
```

**规则**：
```
OR 连接的条件：
- 如果所有列都有索引 → 可以使用索引
- 如果任一列没有索引 → 全表扫描
```

---

### 7️⃣ NOT、!=、<> 导致索引失效 ❌

```sql
-- ❌ 索引失效（大部分情况）
SELECT * FROM user WHERE age != 25;

-- ❌ 索引失效
SELECT * FROM user WHERE age <> 25;

-- ❌ 索引失效
SELECT * FROM user WHERE age NOT IN (20, 25, 30);
```

**例外情况**（可以使用索引）：
```sql
-- ✅ 覆盖索引场景
SELECT age FROM user WHERE age != 25;

-- 说明：查询的列都在索引中，不需要回表
```

```sql
-- ✅ 高选择性场景
SELECT * FROM user WHERE status != 'deleted';

-- 说明：如果 'deleted' 占比很小，优化器可能选择索引扫描
```

---

### 8️⃣ IS NULL / IS NOT NULL

```sql
-- ⚠️ 视情况而定
SELECT * FROM user WHERE age IS NULL;

-- 如果 age 列允许 NULL 且有索引
-- 优化器根据数据分布决定是否使用索引
```

```sql
-- ✅ 可以使用索引
SELECT * FROM user WHERE age IS NOT NULL AND age > 25;

-- 说明：配合其他条件可以使用索引
```

---

### 9️⃣ IN / NOT IN

```sql
-- ✅ IN 可以使用索引
SELECT * FROM user WHERE age IN (20, 25, 30);

-- ❌ NOT IN 索引失效
SELECT * FROM user WHERE age NOT IN (20, 25, 30);

-- ✅ 改写为 NOT EXISTS
SELECT * FROM user u 
WHERE NOT EXISTS (
    SELECT 1 FROM exclude_age e WHERE e.age = u.age
);
```

```sql
-- ⚠️ IN 列表过大可能失效
SELECT * FROM user WHERE id IN (1,2,3,...,1000);

-- 说明：列表过长，优化器可能选择全表扫描
```

---

### 🔟 字符串不加引号 ❌

```sql
-- 假设 code 是 VARCHAR 类型

-- ❌ 索引失效（隐式转换）
SELECT * FROM product WHERE code = 12345;

-- ✅ 正确
SELECT * FROM product WHERE code = '12345';
```

---

### 1️⃣1️⃣ ORDER BY 导致索引失效

```sql
-- 假设索引：idx_name_age (name, age)

-- ✅ 使用索引
SELECT * FROM user ORDER BY name, age;

-- ❌ 索引失效（顺序不一致）
SELECT * FROM user ORDER BY age, name;

-- ❌ 索引失效（跳过左侧列）
SELECT * FROM user ORDER BY age;

-- ❌ 索引失效（排序方向不一致）
SELECT * FROM user ORDER BY name ASC, age DESC;
```

---

### 1️⃣2️⃣ 复合索引中列的顺序

```sql
-- 索引：idx_abc (a, b, c)

-- ✅ 可以使用索引
WHERE a = 1

-- ✅ 可以使用索引
WHERE a = 1 AND b = 2

-- ✅ 可以使用索引
WHERE a = 1 AND b = 2 AND c = 3

-- ❌ 索引失效
WHERE b = 2

-- ❌ 索引失效
WHERE c = 3

-- ⚠️ 部分使用索引
WHERE a = 1 AND c = 3
-- 只使用 a 列索引，c 列不使用
```

## 📊 LIKE 查询详细分析

### 前缀匹配为什么能用索引？

```
索引结构（B+ 树）：
张三丰
张三丰
张无忌
张无忌
张 三
李四
王五

查询：name LIKE '张%'

1. 在索引中找到第一个"张"开头的位置
2. 向后扫描，直到不是"张"开头
3. 返回这些记录的行指针
4. 回表查询完整数据
```

### 后缀匹配为什么不能用索引？

```
索引结构（B+ 树）：
张三丰
李张三
王张三
张 三
李四
王五

查询：name LIKE '%三'

问题：
1. "三"可能出现在任何位置
2. 索引是按前缀排序的，不是按后缀
3. 无法定位起始位置
4. 只能全表扫描
```

### 特殊场景：覆盖索引

```sql
-- 索引：idx_name(name)

-- ✅ 使用覆盖索引（不需要回表）
SELECT name FROM user WHERE name LIKE '张%';

-- ❌ 需要回表
SELECT * FROM user WHERE name LIKE '张%';

-- 说明：覆盖索引性能更好
```

## 🛠️ 优化方案

### 1. 使用前缀索引

```sql
-- 场景：URL、长字符串
-- URL: https://www.example.com/path/to/page

-- ❌ 完整索引浪费空间
CREATE INDEX idx_url ON page(url);

-- ✅ 前缀索引
CREATE INDEX idx_url_prefix ON page(url(30));

-- 查询
SELECT * FROM page WHERE url LIKE 'https://www.example.com%';
```

### 2. 使用全文索引（MySQL 5.7+）

```sql
-- 场景：需要前后模糊匹配
CREATE TABLE article (
    id INT PRIMARY KEY,
    content TEXT,
    FULLTEXT INDEX ft_content (content)
);

-- ✅ 全文搜索
SELECT * FROM article 
WHERE MATCH(content) AGAINST('关键词');
```

### 3. 使用搜索引擎（Elasticsearch）

```
场景：复杂搜索需求
- 前后模糊匹配
- 分词搜索
- 相关性排序

方案：MySQL + ES 双写
- MySQL 存储完整数据
- ES 负责搜索
```

### 4. 使用生成列

```sql
-- 场景：需要函数查询
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    name_upper VARCHAR(100) GENERATED ALWAYS AS (UPPER(name)),
    INDEX idx_name_upper (name_upper)
);

-- ✅ 可以使用索引
SELECT * FROM user WHERE name_upper = 'ZHANGSAN';
```

### 5. 使用 UNION 改写 OR

```sql
-- ❌ 索引失效
SELECT * FROM user WHERE name = '张三' OR age = 25;

-- ✅ 使用 UNION
SELECT * FROM user WHERE name = '张三'
UNION
SELECT * FROM user WHERE age = 25;
```

## 💡 面试加分项

### 扩展 1：索引下推（ICP）

> **索引下推**（Index Condition Pushdown）是 MySQL 5.6 引入的优化：
> - 传统：索引过滤 → 回表 → WHERE 过滤
> - ICP：索引过滤 + WHERE 过滤 → 回表
> 
> **示例**：
> ```sql
> -- 索引：idx_name_age (name, age)
> SELECT * FROM user WHERE name LIKE '张%' AND age > 20;
> 
> -- 无 ICP：
> -- 1. 索引找到所有 name='张%'的记录
> -- 2. 回表查询完整数据
> -- 3. 过滤 age > 20
> 
> -- 有 ICP：
> -- 1. 索引找到 name='张%'的记录
> -- 2. 在索引层过滤 age > 20
> -- 3. 只回表符合条件的记录
> ```

### 扩展 2：覆盖索引

> **覆盖索引**（Covering Index）：
> - 查询的列都在索引中，不需要回表
> - 性能提升显著
> 
> **示例**：
> ```sql
> -- 索引：idx_name_age (name, age)
> 
> -- ✅ 覆盖索引
> SELECT name, age FROM user WHERE name = '张三';
> 
> -- ❌ 需要回表
> SELECT * FROM user WHERE name = '张三';
> ```

### 扩展 3：最左前缀原理

> **为什么联合索引要遵循最左前缀？**
> 
> ```
> 联合索引 (a, b, c) 的排序规则：
> 1. 先按 a 排序
> 2. a 相同的情况下，按 b 排序
> 3. a、b 都相同的情况下，按 c 排序
> 
> 示例数据：
> (1, 1, 1)
> (1, 1, 2)
> (1, 2, 1)
> (1, 2, 2)
> (2, 1, 1)
> (2, 1, 2)
> 
> 查询 WHERE b = 1：
> - (1, 1, 1), (1, 1, 2), (2, 1, 1), (2, 1, 2)
> - 这些数据在索引中不连续
> - 无法使用索引范围扫描
> ```

### 扩展 4：执行计划分析

> **EXPLAIN 关键字段**：
> 
> ```sql
> EXPLAIN SELECT * FROM user WHERE name = '张三';
> ```
> 
> | 字段 | 含义 | 好/坏 |
> |------|------|-------|
> | type | 访问类型 | system > const > eq_ref > ref > range > index > ALL |
> | key | 实际使用的索引 | 有索引 > NULL |
> | rows | 扫描行数 | 越少越好 |
> | Extra | 额外信息 | Using index > Using where > Using temporary > Using filesort |
